# Анализ рисков импорта CSV (40MB)

При работе с файлом такого объема в среде Cloudflare Edge/D1 могут возникнуть следующие сложности:

### 1. Лимиты памяти (Memory Limits)
*   **Проблема:** Чтение всего файла 40МБ в строку (`TextDecoder`) и последующий `.split('\n')` создает огромный массив строк в оперативной памяти. В Edge Runtime лимит обычно составляет 128МБ.
*   **Риск:** Ошибка `Worker exceeded memory limits`.
*   **Решение:** В будущем нужно перейти на потоковый парсинг (streaming), но для 100 строк текущий метод безопасен.

### 2. Таймауты (Execution Time)
*   **Проблема:** 100 итераций, в каждой из которых происходит минимум 2-3 запроса к базе (Upsert Brand, Upsert Category, Create Product), могут занять более 30 секунд.
*   **Риск:** Обрыв соединения до завершения импорта.
*   **Решение:** Использовать `Prisma.$transaction` или пакетную вставку (Batch). Для теста 100 штук должно хватить обычного цикла.

### 3. Кодировка (Encoding)
*   **Проблема:** Если CSV сохранен в Windows-1251 (стандарт Excel в СНГ), кириллица превратится в "кракозябры".
*   **Риск:** Повреждение данных в базе.
*   **Решение:** Сейчас используется UTF-8. Если увидите ошибки в названиях — нужно будет добавить определение кодировки.

### 4. Сложные разделители
*   **Проблема:** Если внутри названия товара есть запятая (например, `"Смеситель, хром"`), простой `.split(',')` разобьет одну колонку на две.
*   **Риск:** Смещение данных по столбцам.

### Как удалить тестовые данные:
Если эксперимент неудачный, выполните в терминале (или попросите меня):
`npx prisma db execute --stdin "DELETE FROM Product WHERE description LIKE '%[EXPERIMENTAL_BATCH_1]%';"`
